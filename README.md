# 客服 FAQ 自动分类脚本 — Code Review & 改进报告

## 📋 项目概述

这是一个"客服 FAQ 自动分类"脚本的 Code Review 项目。原始脚本（`classifier.py`）用于将用户问题自动分配到 6 个客服类别（退款退货、物流查询、账号问题、商品咨询、投诉建议、其他）。本文档记录审查发现的问题、改进方案、评估结果。

## 🔍 1. Code Review：发现的问题

按严重程度排序：

### 🔴 问题 1（严重）：API Key 硬编码

```python
openai.api_key = "sk-proj-abc123def456ghi789jkl012mno345pqr678stu901vwx234"
```

**影响**：
- 代码提交到 Git 仓库后，API Key 完全暴露给所有有仓库访问权限的人
- 泄露后可被他人盗用调用 OpenAI API，造成费用损失
- 撤销泄露的 Key 需要手动到 OpenAI 后台操作

**改进**：从环境变量 `OPENAI_API_KEY` 读取，支持构造函数参数传入，默认不设 Key 时自动回退 Mock 模式。

### 🔴 问题 2（严重）：无输出校验

```python
result = response.choices[0].message.content.strip()
return result
```

**影响**：
- 模型可能返回 `"退款/退货"`、`"退款退货。"`、`"属于退款退货"` 等非标准格式
- 返回 `"退款/退货"` 或 `"投诉/建议"` 时，下游系统无法匹配已定义的类别名称
- 模型可能在类别名后附加解释文字，导致整条结果不可用

**改进**：加入三层标准化兜底函数 `_normalize_category()`：
1. 精确匹配（直接命中）
2. 模糊包含匹配（字符串包含关系）
3. 关键词启发式计分（关键词命中数汇总）

### 🔴 问题 3（严重）：无错误处理

```python
response = openai.chat.completions.create(...)
```

**影响**：
- 网络超时、API 限流（Rate Limit）、服务器异常直接抛出未捕获异常
- 整批处理全部失败，已处理的中间结果丢失
- 生产环境零健壮性

**改进**：加入指数退避重试机制（最多 3 次，`delay × 2^n`），捕获所有异常，单条失败不影响整体流程。

### 🟡 问题 4（高）：Prompt 设计粗糙

当前 prompt：
```
你是一个客服分类助手。请对以下用户问题进行分类。
分类类别：退款退货、物流查询、账号问题、商品咨询、投诉建议、其他
用户问题：{question}
请直接回复分类结果，只回复类别名称。
```

**影响**：
- **无 System Prompt**：所有指令写在 User Message 中，没有利用 System/User 角色分离
- **无分类定义**：模型不知道每个类别的具体含义，靠关键词和语感猜测
- **无分类规则**：没有"主诉求优先"、"含投诉内容→投诉建议"等业务规则
- **无 few-shot 示例**：没有示例示范，模型在处理边界情况时表现不稳定
- 导致以下典型错误：
  - `"建议你们增加夜间配送选项"` → 看到"配送"误判为物流查询（实际：投诉建议）
  - `"买了三天就坏了，什么破质量"` → 看到"质量"误判为商品咨询（实际：投诉建议）
  - `"退货流程太麻烦，搞不懂怎么操作"` → 看到"退货"误判为退款退货（实际：投诉建议）

**改进**：完全重构 Prompt（详见第 2 节）。

### 🟡 问题 5（高）：无日志系统

```python
print(f"分类完成，共处理 {len(results)} 条问题")
```

**影响**：
- 只有一行输出，完全无法排查单条失败
- 无法追踪调用耗时、API 错误信息
- 生产环境无法定位问题

**改进**：使用 `logging` 模块，每处理一条输出进度和分类结果，支持配置日志级别。

### 🟢 问题 6（中）：无 Mock 模式

**影响**：
- 每次开发调试都需要调用真实 API，产生费用
- 无法离线测试和评估

**改进**：加入 `mock=True` 模式，基于关键词规则模拟分类行为，零成本跑通全流程。

### 🟢 问题 7（低）：缩进格式

原始文件使用 1 个空格作为缩进单位，而非 Python 社区标准的 4 个空格。语法上虽然合法，但可读性差，且在 `batch_classify` 函数中存在 `with` 块体缩进不正确的语法问题。

**改进**：V2 版统一使用 4 空格标准缩进。

## 📝 2. Prompt 改进设计

### V1 Prompt

```
你是一个客服分类助手。请对以下用户问题进行分类。
分类类别：退款退货、物流查询、账号问题、商品咨询、投诉建议、其他
用户问题：{question}
请直接回复分类结果，只回复类别名称。
```

全部写在 User Message 中，无角色分离、无定义、无规则、无示例。

### V2 Prompt 设计

**System Prompt**（角色定义 + 分类体系 + 分类规则 + 示例）：

```
你是一位专业的电商客服分类助手。你的任务是将用户的提问准确归类到相应的客服组。

## 分类体系

1. **退款退货** — 用户要求退款、退货、换货，或咨询退款进度、退货流程等
2. **物流查询** — 用户询问包裹位置、配送状态、快递信息、地址修改等物流相关问题
3. **账号问题** — 用户遇到登录、密码、账号安全、信息修改等账号相关问题
4. **商品咨询** — 用户询问商品信息、规格、库存、价格、材质、使用方法等
5. **投诉建议** — 用户对服务或商品质量不满、提出投诉或改进建议
6. **其他** — 不属于以上任何类别的表述，包括闲聊、问候、无意义字符等

## 分类规则

1. 主诉求优先：如果问题涉及多个类别，以用户的主要诉求为准
2. 退款进度 → 退款退货：询问"退款什么时候到账"等属于退款类
3. 含投诉内容 → 投诉建议：即使包含了具体问题描述，只要核心诉求是表达不满或建议，就归入投诉建议
4. 纯问候/无意义 → 其他

## 示例（few-shot）

用户问题：七天无理由退货怎么退
类别：退款退货

用户问题：快递到哪了，三天了还没到
类别：物流查询

用户问题：我忘记密码了，怎么找回
类别：账号问题

用户问题：这个耳机支持降噪吗
类别：商品咨询

用户问题：你们客服态度太差了，我要投诉
类别：投诉建议

用户问题：你好
类别：其他

## 输出要求

只输出一个类别名称，不要包含任何其他文字、标点或解释。
```

**User Message**（极简）：

```
用户问题：{question}
```

### 改动对比

| 项目 | V1 | V2 |
|------|-----|-----|
| 角色分配 | User Message 包含全部指令 | System Prompt 定义角色和规则，User Message 仅放问题 |
| 分类定义 | 仅列出 6 个类别名称 | 每个类别配详细定义和典型场景 |
| 分类规则 | 无 | 4 条明确规则（主诉求优先、退款界定、投诉界定、无意义过滤） |
| 示例 | 无 | 6 条 few-shot 示例覆盖所有类别 |
| 输出约束 | "只回复类别名称" | 更严格约束 + `max_tokens=20` |

3. **few-shot 示例**：提供了"客服态度太差→投诉建议"的参考

## 🛠 4. 工程化改进（4 项）

### 4.1 安全加固：API Key 从环境变量读取

```python
key = api_key or os.environ.get("OPENAI_API_KEY")
if not key:
    raise ValueError("未提供 API Key...")
```

- 支持构造函数参数和 `OPENAI_API_KEY` 环境变量两种方式
- 未提供 Key 时切入 Mock 模式，开发调试零成本

### 4.2 输出标准化（三层兜底）

```python
def _normalize_category(text: str) -> str:
    # 第一层：精确匹配
    if text in CATEGORIES:
        return text
    # 第二层：模糊包含匹配
    for cat in CATEGORIES:
        if cat in text or text in cat:
            return cat
    # 第三层：关键词计分
    scores = {cat: 0 for cat in CATEGORIES}
    for cat, keywords in keywords_map.items():
        for kw in keywords:
            if kw in text:
                scores[cat] += 1
    return max(scores, key=scores.get) or "其他"
```

### 4.3 错误重试与日志

```python
for attempt in range(1, MAX_RETRIES + 1):
    try:
        response = self._client.chat.completions.create(...)
        return result
    except Exception as e:
        logger.warning(f"第 {attempt}/{MAX_RETRIES} 次失败: {e}")
        if attempt < MAX_RETRIES:
            time.sleep(RETRY_BASE_DELAY * 2 ** (attempt - 1))
logger.error(f"所有重试失败，问题: {question}")
return "其他"  # 兜底
```

- 最多 3 次重试，指数退避（1s → 2s → 4s）
- 单条失败仅影响本条，兜底返回"其他"

### 4.4 Mock 模式 + 评估脚本

```bash
# V2 分类 + 评估（Mock）
python3 classifier_v2.py test_samples.json results_v2.json --mock --evaluate

# V1 vs V2 对比评估
python3 evaluate_runner.py
```

- Mock 模式基于精心设计的关键词规则，模拟改进后的分类行为
- `evaluate_runner.py` 自动对比 V1 vs V2 准确率，输出详细差异
- 结果保存为 `results_v1.json` / `results_v2.json`
